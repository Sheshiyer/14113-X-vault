### **Orchestration Principles**

1. **Stateless Execution** - Each engine calculation is independent
2. **Parallel Processing** - Multiple engines can execute simultaneously when inputs are ready
3. **Deterministic Results** - Same inputs always produce same outputs (for non-interpretive engines)
4. **Graceful Degradation** - If one engine fails, others continue
5. **Cache-First Strategy** - Check cache before computing

```python
# Pseudo-code for workflow orchestration

async def execute_natal_blueprint_workflow(birth_data):
    # Step 1: Parse workflow definition
    workflow = load_workflow("natal-blueprint")
    
    # Step 2: Prepare engine calls (parallel)
    tasks = [
        numerology_engine.calculate_life_path(birth_data),
        human_design_engine.generate_chart(birth_data),

        "engine_results": results,
        "synthesis": synthesis,
        "narrative": narrative,
        "witness_prompts": witness_prompts,

    }
```

- Essential for Vimshottari Dasha calculations
- Gene Keys depend on accurate HD calculations

**Alternative Considered:** Astronomy.js, Astronomia  
**Why Rejected:** Lower precision, incomplete planetary data

#### **2. Stateless Engine Architecture**

**Decision:** All engines are stateless, deterministic functions.

**Rationale:**
- Easier to cache (same inputs = same outputs)
- Easier to test
- Easier to scale horizontally
- No need for engine-specific databases
- Aligns with "mirror, not advisor" philosophy

**Trade-off:** Cannot "learn" from user feedback, but this is INTENTIONAL - we want consistent mirrors, not adaptive systems that could create dependency.

#### **3. Redis for Aggressive Caching**

**Decision:** Cache all deterministic calculations with Redis.

**Rationale:**
- Human Design chart for "1990-01-15 14:30 New York" never changes
- Same for Numerology, Biorhythm, Vimshottari
- Astronomical calculations are computationally expensive
- 85%+ cache hit rate achievable (validated in testing)

**Cache Strategy:**
```python
cache_key = f"hd:{birth_date}:{birth_time}:{lat}:{lon}"
ttl = 365 * 24 * 60 * 60  # 1 year (effectively permanent)
```

#### **4. No Real-Time Biofield (Initially)**

**Decision:** Biofield engine requires PIP device data upload, not real-time streaming.

**Rationale:**
- PIP devices are specialized hardware (not everyone has)
- Real-time streaming adds significant complexity
- Upload + analysis model simpler for MVP
- Can add real-time in phase 3

**Future:** WebSocket/SSE for real-time if demand exists

#### **5. Anti-Prescription AI Prompts**

**Decision:** If using LLM for narrative synthesis, system prompts enforce anti-prescription.

**Example System Prompt:**
```
You are a seasoned cartographer helping someone read their own map.
NEVER prescribe actions. ALWAYS reflect patterns back as questions.

WRONG: "You should focus on..." 
RIGHT: "This pattern suggests... What do YOU see in this?"

WRONG: "Do this next..."
RIGHT: "These converge here. Where does YOUR attention go?"
```

## ğŸ” **Core Architectural Constraints**

### **1. Anti-Dependency Architecture**

**Principle:** Every feature must prevent dependency, not create it.

**Implementation:**
- Phased engine access (not all at once)
- Witness Agent prompts after every result
- No "recommended next steps" features
- Somatic pacing enforced (breath timers)
- Success metric = user independence, not engagement time

### **2. Privacy-First Data Architecture**

**Principle:** User birth data is sensitive; minimize retention.

**Implementation:**
- Birth data encrypted at rest
- Option to calculate without storing (stateless API)
- No third-party analytics on birth data
- User can delete all data at any time
- Engine results NOT stored (recalculated from cache/source)

### **3. Calculation Transparency**

**Principle:** Users should be able to verify calculations.

**Implementation:**
- Show calculation methodology in UI (expandable)
- Provide references to source algorithms
- Allow export of raw calculation data
- Compare with professional software (HD, Vimshottari)

### **4. Graceful Degradation**

**Principle:** If one engine fails, others continue.

**Implementation:**
```javascript
try {
  const hdResult = await calculateHumanDesign(birthData);
} catch (error) {
  console.error("HD calculation failed:", error);
  hdResult = { error: "Calculation unavailable", reason: error.message };
}
// Continue with other engines regardless
```
